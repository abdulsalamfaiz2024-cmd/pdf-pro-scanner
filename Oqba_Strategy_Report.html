02
Architecture & Tooling Strategy

ONE CRITICAL FLAW
IN YOUR STACK.

Your framework choice is sound. Your scanner library choices are acceptable. But there is a dangerous gap in your PDF manipulation plan that will cause you to hit a wall at Hour 12.

⚑ Critical Flaw Identified: The PDF Library Gap

The Flutter pdf package listed in your plan is a PDF creation library — it builds PDFs from scratch using widgets. It cannot open, read, or manipulate existing PDF files. This means your Merge and all future PDF tools will fail entirely at Hour 12 when you try to merge real files the user scanned. This single error could eliminate you from the competition.

Revised Stack Recommendation
Layer Original Plan Verdict Recommended Alternative Reason
Framework Flutter (Dart) Keep Flutter — confirmed Native performance, single codebase, rich package ecosystem. Correct choice.
Scanner SDK cunning_document_scanner Upgrade google_mlkit_document_scanner Wraps Google ML Kit's on-device Document Scanner API. Handles edge detection, perspective correction, and image enhancement natively. More actively maintained, better Android consistency, and it's 100% offline by design since ML Kit models ship bundled.
PDF Manipulation pdf package Replace syncfusion_flutter_pdf Can load, read, merge, split, and write existing PDF files — exactly what you need. The community license is free. This is the correct library for PDF manipulation vs. the pdf package which only creates PDFs from widgets.
State Management Provider or Riverpod Simplify setState only In a 24-hour window, setting up Riverpod providers, notifiers, and async states is a time sink. Use simple setState for this MVP. You can refactor after the demo. Don't architecture-astronaut yourself out of a win.
File System path_provider + file_picker Keep As planned — confirmed This combination is the industry standard and well-tested. No changes needed.
React Native Alt. pdf-lib Keep (if RN) pdf-lib — confirmed If you go the React Native route, pdf-lib is the correct choice. It handles real PDF manipulation in JavaScript. The RN scanner plugin choice is acceptable but test it on a physical device on Day 1, not Hour 6.
Golden Architecture Rule for Offline Apps

Structure your app around a local file-system contract from Hour 1. Define one standard path where all scanned and managed PDFs live (e.g., AppDocuments/Oqba/). Every feature reads from and writes to this single location. This prevents the silent failure where your scanner saves to a temp cache but your PDF merger looks in the documents folder and finds nothing — a common and devastating demo failure.

03
Risk Mitigation — Crash Prevention

THE FIVE THINGS
THAT WILL KILL
YOUR DEMO.

Offline image and PDF processing has specific, well-known failure points. Here are the five architectural threats ranked by probability and the strategic rule to neutralize each one before your live demo.

Memory Exhaustion on Large PDFs High Risk

Loading a multi-page scanned PDF entirely into memory to merge it is the most common crash vector in PDF apps. A 20-page scanned document at high resolution can be 40–80MB. Loading two of them simultaneously for a merge operation can easily exceed device memory limits, causing an immediate silent kill of your app.

Strategic Rule: Never load the full byte contents of a PDF into a single in-memory variable. Use your PDF library's stream-based or page-by-page APIs. Process pages sequentially and release each one. For your demo, control the narrative — use a pre-prepared PDF of 3–5 pages. There is no rule that says the judge's file must be used.
Camera & Storage Permissions Crash High Risk

On Android 13+ and iOS 14+, permissions are granular, revocable, and dialog-based. If your app requests camera permission after it has already been denied once, it will silently fail or show a broken UI. This is the single most common cause of demo failures in scanner apps.

Strategic Rule: Build a dedicated "Permissions Gate" screen. Before launching the camera, always check the permission status first. If denied, show a clear in-app message with a button that opens the device's Settings app — never leave the user on a blank screen. Test this on a physical device with permissions denied. Test it on two physical devices: one Android, one iOS.
Plugin Version Conflicts at Build Time Medium Risk

Flutter's pub.dev ecosystem has dependency conflicts. Adding a new scanner plugin often pins certain versions of Kotlin or Gradle that conflict with other plugins. This can surface as a build error at Hour 6 that costs you 2–3 hours to debug.

Strategic Rule: Install ALL plugins in Hour 0–1, before writing a single line of UI code. Run a test build immediately. If the build is green, commit it as your "safe baseline." This way you discover conflicts before you've written code that depends on them, and rolling back is painless.
File Path Inconsistency (iOS vs Android) Medium Risk

iOS sandboxing and Android scoped storage use fundamentally different path structures. A file path returned by your scanner plugin on Android is often a content URI (e.g., content://media/...), not a real file path. Passing this directly to your PDF library will cause a "file not found" crash that looks confusing and is hard to diagnose under pressure.

Strategic Rule: Immediately after scanning or picking any file, copy it to your app's internal documents directory and work with that internal copy exclusively. Never pass external URIs or cache paths to your PDF processing functions. This single rule prevents 90% of cross-platform file handling crashes.
UI Freezing During Processing Low Risk (High Impact)

Performing PDF manipulation on the main UI thread will freeze the app for 1–3 seconds depending on file size. The app won't crash, but a frozen UI during a live demo looks like a crash to every person watching.

Strategic Rule: Wrap every PDF operation in an async function and show a loading indicator before it starts. In Flutter, use a simple compute() call or an isolate to move the heavy work off the main thread. The loading indicator makes processing feel intentional and fast — not broken.
04
The "Wow" Factor — High-ROI UX

THREE TRICKS THAT
MAKE JUDGES LEAN IN.

These are not cosmetic features. They are psychological tools. Each one takes under 30 minutes and will be remembered far more than a fourth PDF tool.

01
The Speed Receipt
⏱ Implementation: ~20 minutes

After every operation (scan, merge, whatever) display a small "operation complete" banner that shows the processing time in milliseconds — e.g., "Merged in 340ms · Offline". This does two things simultaneously: it proves your app is fast and it proves it is offline. Judges who might not fully grasp the technical difficulty of offline PDF processing will see "Offline" stamped on every result and understand instantly.

Psychological trigger: Concrete numbers signal engineering rigor. "Fast" is vague. "340ms" is a proof point.
02
Haptic Scan Shutter
⏱ Implementation: ~10 minutes

Trigger a medium haptic feedback pulse at the exact moment a document is captured. This is a single function call in both Flutter and React Native. Judges watching a live demo will feel the app respond to the physical action of scanning. It transforms a software button press into a tactile, physical experience — exactly like a professional camera shutter. It makes the app feel premium without any visual effort.

Psychological trigger: Haptics bypass visual attention. The judge feels the quality before they see it. It creates a memory.
03
The Privacy Heartbeat
⏱ Implementation: ~25 minutes

Add a subtle, permanently visible indicator in your app's top bar — a small icon or a single-word label that reads "Offline" or "Private" with a soft pulsing animation. This keeps your core differentiator visible at all times. When you're pitching, you don't have to tell the judges the app is private — they can see it on screen the entire time. The feature markets itself passively and continuously throughout the demo.

Psychological trigger: Repetition and visibility. The message "this is private" is reinforced every second the app is on screen, not just when you mention it verbally.
One Additional Micro-Touch Worth Noting

Replace generic loading spinners with a skeleton loader on your home file list. When the app loads and reads the local file system, instead of a spinner, show ghost-gray placeholder cards in the shape of file items. This takes 30 extra minutes but makes the app feel dramatically more polished and production-ready — the difference between a hackathon prototype and a real product.

05
Revised Execution Blueprint

YOUR CORRECTED
24-HOUR SCHEDULE.

The original timeline has a fatal flaw: it leaves integration and wiring to Hours 14–18, which is too late. The revised plan front-loads all integration risk so you discover problems early, while you still have time to recover.

Hr 0–1
Phase 1 — Foundation
Create project. Install ALL plugins immediately (scanner, PDF library, file picker, path provider). Run a test build on a physical device. Commit as "Baseline Build." Do not write any UI yet.
⚑ Critical: If your build is broken here, you fix it now — not at Hour 12 when you're exhausted.
Hr 1–3
Phase 2 — Scanner Integration
Integrate the scanner plugin first — before UI. Get the camera open and a captured image saved to the app's internal directory. This is your highest-risk integration and must be validated early. Once it saves an image, immediately convert it to a PDF and confirm the file exists on disk.
Hr 3–5
Phase 3 — PDF Logic Core
Build and test the Merge function using syncfusion_flutter_pdf. Test it with two real dummy PDF files you pre-create. Confirm the output file is valid and can be opened. This must work perfectly before you build any UI around it.
⚑ Critical: Both scanner and PDF merge must be working in isolation by Hour 5. If either is broken, you have time to switch libraries or adapt.
Hr 5–9
Phase 4 — UI Shell
Build the UI with hardcoded placeholder data. Home screen with file list (skeleton loaders), Tools grid showing only Scan and Merge, the "About / Privacy" screen. Keep the design focused and premium. Dark mode, clean typography, generous spacing.
Hr 9–13
Phase 5 — Wiring
Connect the UI to your working logic. Scan button → scanner → save to app directory → refresh home screen file list. Merge button → file picker → select two PDFs → merge function → save output → show success with file size and processing time.
Hr 13–17
Phase 6 — Polish & Wow Factors
Add haptic feedback on scan. Add the processing time + "Offline" badge to all success states. Add the permanent "Offline" indicator to the top bar with its subtle animation. Add permission handling flows. Add error toasts for edge cases.
⚡ This phase is where you transform a functional app into a winning one. Protect this time aggressively.
Hr 17–21
Phase 7 — Device Testing
Test the full demo loop end-to-end on a physical device a minimum of ten times without touching the code. Use real documents — a receipt, a page from a book, a printed form. Fix only crashes and visual glitches. Feature additions are completely banned in this phase.
Hr 21–24
Phase 8 — Demo Prep
Build the release APK. Pre-load two documents onto the demo device. Rehearse the 60-second pitch five times. Prepare a backup device with the app installed. Sleep or rest if possible — a sharp presenter beats a polished app.

A demo is not a tour of features. It is a story with a villain, a hero, and a resolution. Your villain is surveillance capitalism and cloud dependency. Your hero is Oqba. Here is how you tell that story in exactly 60 seconds.

Three Cardinal Demo Rules
  • 01 Never ask permission to demo. Do not say "so I'm going to show you how to scan something." Just scan something. Confidence in action communicates more than confidence in words.
  • 02 Pre-load a document on the device. Have one scanned PDF already saved in the app before you walk in. This gives you a guaranteed fallback — if the live scan fails for any reason (bad lighting, camera freeze), you can pivot instantly to "I've already scanned one here, let me show you the merge" without breaking the demo's momentum.
  • 03 Say "offline" every time the app says "offline." Each time the Offline indicator is visible and you're speaking, say the word "offline" out loud. Repetition is a persuasion tool. By the end of 60 seconds, the word should be anchored in every judge's mind as the defining characteristic of Oqba.
